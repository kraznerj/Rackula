## Overview

- Authentication architecture (Better Auth with stateless sessions)
- Generic OIDC support (any compliant IdP)
- Session management (cookie-based, 12-hour TTL, 6-hour refresh)
- Read-only access for unauthenticated users

## Prerequisites

- OIDC-compliant identity provider (Authentik, Authelia, Keycloak, etc.)
- HTTPS-enabled Rackula deployment (required for secure cookies)
- Access to IdP admin console

## OIDC Configuration

### Step 1: Configure Your Identity Provider

Provide specific setup instructions for common IdPs:

**Authentik:**

- Create new OAuth2/OIDC Provider
- Set Redirect URIs: `https://your-rackula-domain.com/auth/callback`
- Set Scopes: `openid`, `profile`, `email`
- Note Client ID and Client Secret
- Copy Issuer URL from Provider details

**Authelia:**

- Add client to `configuration.yml` under `identity_providers.oidc.clients`
- Set `client_id`, `client_secret`, `redirect_uris`
- Set `scopes: ['openid', 'profile', 'email']`
- Issuer URL is `https://your-authelia-domain.com`

**Keycloak:**

- Create new Client in Realm
- Set Access Type: confidential
- Set Valid Redirect URIs: `https://your-rackula-domain.com/auth/callback`
- Enable Standard Flow
- Copy Client ID and Client Secret from Credentials tab
- Issuer URL: `https://your-keycloak-domain.com/realms/{realm-name}`

### Step 2: Configure Rackula Environment Variables

Document required environment variables:

```bash
# OIDC Provider Configuration
RACKULA_OIDC_ISSUER=https://your-idp.com/...
RACKULA_OIDC_CLIENT_ID=your-client-id
RACKULA_OIDC_CLIENT_SECRET=your-client-secret
RACKULA_OIDC_REDIRECT_URI=https://your-rackula-domain.com/auth/callback

# Session Configuration (optional, defaults shown)
RACKULA_AUTH_SESSION_MAX_AGE_SECONDS=43200  # 12 hours
RACKULA_AUTH_SESSION_UPDATE_AGE_SECONDS=21600  # 6 hours

# Cookie Security (production defaults)
RACKULA_AUTH_SESSION_COOKIE_SECURE=true  # HTTPS only
RACKULA_AUTH_SESSION_COOKIE_SAMESITE=lax  # CSRF protection
```

### Step 3: Verify Configuration

Testing checklist:

1. Visit `https://your-rackula-domain.com/` (should show app, read-only mode)
2. Visit `https://your-rackula-domain.com/auth/login` (redirects to IdP)
3. Complete IdP login flow (redirects back to `/auth/callback`)
4. Session cookie set (check browser DevTools → Application → Cookies)
5. Access protected routes (can save layouts)

## Security Hardening

### Session Security

- Keep session TTL short (12 hours default, reduce if needed)
- Use HTTPS in production (enforces Secure cookie flag)
- Set SameSite=Lax (default, prevents CSRF)
- Rotate session secret regularly (not implemented yet, note for future)

### OIDC Security

- Always use HTTPS for OIDC issuer (Better Auth validates TLS)
- Store client secret securely (Docker secrets, not environment variables in production)
- Validate redirect URI matches exactly (Better Auth validates this)
- Use minimal scopes (openid, profile, email only)

### Production Deployment

- Mount secrets via Docker secrets: `/run/secrets/oidc_client_secret`
- Never commit `.env` files with real credentials
- Use environment-specific configuration (dev vs prod)
- Monitor auth event logs (future enhancement, not in v0.9.0)

## Docker Secrets Integration

Example docker-compose.yml snippet:

```yaml
services:
  api:
    secrets:
      - oidc_client_secret
    environment:
      RACKULA_OIDC_CLIENT_SECRET_FILE: /run/secrets/oidc_client_secret

secrets:
  oidc_client_secret:
    file: ./secrets/oidc_client_secret.txt
```

## Troubleshooting

### "Authentication is enabled, but login/callback handlers are not implemented yet"

- This error should no longer occur after Phase 1
- If seen, verify Better Auth routes are mounted at `/auth/*`

### OIDC callback fails with "Invalid redirect URI"

- Verify RACKULA_OIDC_REDIRECT_URI matches IdP configuration exactly
- Check for trailing slashes (should NOT be present)
- Ensure protocol matches (https vs http)

### Sessions expire immediately

- Check RACKULA_AUTH_SESSION_MAX_AGE_SECONDS is set correctly
- Verify browser accepts cookies (check DevTools)
- Ensure Secure flag matches deployment (https required in production)

### "Unauthorized" when accessing layouts

- Verify session cookie is present (DevTools → Application → Cookies)
- Check session has not expired (12-hour default TTL)
- Confirm auth middleware is configured correctly

## Limitations (v0.9.0)

Current implementation does NOT support:

- Server-side session revocation (stateless cookies cannot be revoked until expiry)
- Auth event logging (planned for v2)
- Local username/password authentication (planned for v2)
- Multi-factor authentication (delegated to IdP)
- User management UI (manage via IdP)

For instant session revocation, upgrade to database-backed sessions (see MIGRATION.md - not yet written).
